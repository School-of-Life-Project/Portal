î‡ Creator's Guide

âš  It is highly recommended that you read all preceding sections of the manual before reading this section.

This guide is intended for those who wish to create resources for Portal. This is a somewhat involved process, and a degree of specialized technical knowledge will be required.

You will need a basic understanding of the following concepts:

  • Tom's Obvious Minimal Language (TOML)
  • Universally Unique Identifier (UUID)
  • JSON Schema
  • EPUB e-Book format
    • EPUB v3.2 specification
    • Open Publication Structure v2 specification (only used by EPUB v2 documents)
    • Extensible Markup Language (XML)
    • Hypertext Markup Language (HTML)
    • OCF abstract container (EPUB-specific file hierarchy)
    • EPUB navigation document
    • Navigation Center eXtended (only used by EPUB v2 documents)
  • ZIP archive format

You will need the following digital tools:

  • A file archiver, such as PeaZip
  • A code editor, such as Visual Studio Code
    • A plugin to validate TOML files against a JSON schema (recommended)
  • A specialized ePub editor, such as Sigil (recommended)

Throughout the rest of this section, Portal's user data folder will be referred to as $APPDATA. Paths are case-sensitive (assuming the underlying filesystem is) unless otherwise specified.

â„¹ When using UUIDs in filesystem paths, the UUID must be represented in lowercase hexadecimal form, without separators (example: 79a55c4e046343039424684764eccd8a). When referring to representations of UUIDs in filesystem paths, the syntax {UUID} will be used.

î‰§ Resource Schema

Portal preserves backwards-compatibility for documented behavior. Updating Portal should not break existing resources â€” if this happens, it's a bug. However, resources designed for newer versions of Portal may fail to load on older versions.

ğŸ“š Courses

Courses are folders containing a TOML file named course.toml at their root. This file, adhering to the $APPDATA/Resource Schema/Course.json schema, contains a set of one or more textbooks (in EPUB format).

All files or folders referenced by the course.toml must be contained within the Course folder. Files or folders within the Course folder which are not referenced by the index are ignored.

Courses are loaded from the path $APPDATA/{UUID}/. On the ğŸ« Home page, only active courses are loaded. In the ğŸ“– Viewer, only the Course being viewed is loaded. In the îŠ Course Navigator, all Courses are loaded, regardless of their active status.

ğŸ“• Textbooks

â„¹ This section assumes that you have an existing textbook, in EPUB format, that you would like to include in a Course. Converting textbooks into the EPUB format, or creating EPUB documents from scratch, is out of scope for this document.

All textbooks referenced by the course.toml must be unpacked EPUB files (the EPUB container must be a folder, not a file). Existing EPUBs can be unpacked by changing their extension to .zip and then extracting the archive. Review the Course schema for a list of supported EPUB versions.

In order for a textbook to be completable, a list of user-completable chapters must be included in the configuration for that textbook. Each referenced section in the chapter must match a valid href from the EPUB's Table of Contents.

â„¹ Only make chapters or sections completable when their completion is meaningful. Sections included for reference or as supplementary material should not be marked as completable.

âš  In order to ensure each textbook within your course is completable, you must test them within the Portal app. Referencing invalid sections will not result in an error.

The ğŸ“– Viewer is the only part of the application which loads Course textbooks (all other areas of the app will only load the course.toml). It will only load the textbook being actively viewed.

The ğŸ« Home page and îŠ Course Navigator are useful areas to check the completion of a Course.

ğŸ—ºï¸ Course Maps

Course Maps are TOML files, adhering to the $APPDATA/Resource Schema/CourseMap.json schema, which contain a user-facing dependency tree of Courses (referenced by UUID).

Course Maps are loaded from the path $APPDATA/User Resources/{UUID}.toml when using the îŠ Course Navigator. An invalid Course Map will result in an immediate error. However, a Course Map which references a non-existent Course will only produce an error once the user attempts to load the Course Map.

î‰  Packaging and Sharing Resources

âš  You should always test all aspects of your resources in the Portal app before packaging and sharing them. Validating your resource against the relevant schema will not catch all possible mistakes.

âš  If you plan on publically distributing a Course, a basic understanding of copyright law is required. Legal guidance is out of scope for this document.

Resources are packaged as unencrypted ZIP archives, which are loaded from the path $APPDATA/User Resources/*.zip (extension is not case sensitive). The Portal app then automatically extracts all detected ZIP archives in the resource folder into $APPDATA/User Resources/, overwriting any files which already exist. Decompression will not be performed recursively, so putting ZIP archives within other ZIP archives will lead to inconsistent behavior. Once the decompression process is successfully completed, the ZIP archive will be deleted from the folder.

â„¹ Unlike Courses or Course Maps, archive filenames do not convey information to the app. You should use these filenames to inform the end user of the archive's contents.

ZIP archives can be uncompressed, or they can be compressed with one of the decompression algorithms supported by the app: DEFLATE, DEFLATE64, BZIP2, ZStandard, LZMA, and XZ.

This form of packaging allows for a substantial degree of flexibility. Courses could be packaged as individual files, or multiple Courses could be packaged together into the same archive. Course Maps could be shared directly as TOML files, or they could be packaged together with the Courses they reference. However, it is important to keep in mind the needs and expectations of non-technical users when determining how to package your resources.

â„¹ Sets of Courses which are intended to be completed together should be packaged together as one archive. If this is not feasible, use multiple archives with filenames indicating their relation (example: My Course Collection (Part 1 of 3).zip).

â„¹ Courses and Course Maps should be distributed together, as one archive containing all relevant resources. Avoid distributing Course Maps separately from Courses.
