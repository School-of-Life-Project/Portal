ğŸ“š Courses

Courses are folders containing a TOML file named course.toml at their root. This file, adhering to the $APPDATA/Resource Schema/Course.json schema, contains a set of one or more textbooks (in EPUB format).

All files or folders referenced by the course.toml must be contained within the Course folder. Files or folders within the Course folder which are not referenced by the index are ignored.

Courses are loaded from the path $APPDATA/User Resources/{UUID}/. On the ğŸ« Home page, only active courses are loaded. In the ğŸ“– Textbook Viewer, only the Course being viewed is loaded. In the îŠ Course Navigator, all Courses are loaded, regardless of their active status.

ğŸ“• Textbooks

â„¹ This section assumes that you have an existing textbook, in EPUB format, that you would like to include in a Course. Converting textbooks into the EPUB format, or creating EPUB documents from scratch, is out of scope for this document.

All textbooks referenced by the course.toml must be unpacked EPUB files (the EPUB container must be a folder, not a file). Existing EPUBs can be unpacked by changing their extension to .zip and then extracting the archive. Review the Course schema for a list of supported EPUB versions.

In order for a textbook to be completable, a list of user-completable chapters must be included in the configuration for that textbook. Each referenced section in the chapter must match a valid href from the EPUB's Table of Contents.

â„¹ Only make chapters or sections completable when their completion is meaningful. Sections included for reference or as supplementary material should not be marked as completable.

The ğŸ“– Textbook Viewer is the only part of the application which loads Course textbooks (all other areas of the app will only load the course.toml). It will only load the textbook being actively viewed.

The îŠ Course Navigator is useful for checking the completion of a Course. Like the ğŸ« Home page, you can click on textbooks within a displayed ğŸ“š Course to open the ğŸ“– Textbook Viewer.

î‰  Packaging and Sharing Resources

âš  You should always test all aspects of your resources in the Portal app before sharing them. Validating your resource against the relevant schema will not catch all possible mistakes.

âš  If you plan on publically distributing a Course, a basic understanding of copyright law is required. Legal guidance is out of scope for this document.

Resources are packaged as unencrypted ZIP archives, which are loaded from the path $APPDATA/User Resources/*.zip (extension is not case sensitive). The Portal app automatically extracts the contents of all detected archives into $APPDATA/User Resources/, overwriting any files which already exist. Decompression will not be performed recursively, so archives should not be put inside other archives. Once the decompression process is successfully completed, the archive will be deleted from the folder.

â„¹ Unlike Courses or Course Maps, archive filenames do not convey information to the app. You should use these filenames to inform the end user of the archive's contents.

ZIP archives can be uncompressed, or they can be compressed with any of the algorithms supported by the app: DEFLATE, DEFLATE64, BZIP2, ZStandard, LZMA, and XZ.

Sets of Courses which are intended to be completed together should be packaged together into one archive. If this is not feasible, use multiple archives with filenames indicating their relation (example: My Course Collection (Part 1 of 3).zip). Courses and Course Maps should be distributed together, as one archive containing all relevant resources. Do not distribute Course Maps separately from Courses.
